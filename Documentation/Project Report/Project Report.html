Project Report: MST Visualization Project

Introduction

The MST Visualization Project is a Python-based desktop application that visualizes the computation of Minimum Spanning Trees (MSTs) using Kruskals and Prims algorithms. Built with PyQt5 for the GUI, NetworkX for graph operations, and Matplotlib for visualization, it allows users to generate random or custom graphs, assign weights, run MST algorithms with animations, and view results in a comparison table. This report provides an overview of the project, its data structures, algorithm components, user interface design, and experimental results.

Project Details

The project aims to provide an interactive tool for visualizing MST algorithms, suitable for education or analysis. Key features include:

  • Generate random graphs (527 nodes by default) or custom graphs (1500 nodes).
  • Assign random weights or specify custom weights for edges.
  • Run Kruskals or Prims algorithm with animation (0.25s/step for 27 nodes, instant for >27).
  • Display results (MST edges, runtime, weights) in a comparison table.
  • Reset the application to start over.

The codebase is modular, with 13 files (e.g., mst_project.py, gui.py, mst_animation.py, combined_metrics_chart.py) each handling a specific responsibility, as detailed in the Code Documentation (Code Documentation.html).

Main Data Structures

The project uses the following data structures to manage graphs, algorithms, and state:

1. Graph Representation

  • Structure: NetworkX Graph (undirected, adjacency list).
  • Files: graph_manager.py, mst_algorithms.py.
  • Role: Stores nodes (integers) and weighted edges (e.g., graph[u][v]['weight']).
  • Usage: Graph generation, MST computation, visualization.
  • Details: Space complexity is O(V + E), where V is nodes and E is edges. Random graphs have ~1.5V - 1 edges (e.g., V = 100, E 149).

2. Union-Find (Disjoint Set)

  • Structure: Lists (parent, rank).
  • File: mst_algorithms.py.
  • Role: Detects cycles in Kruskals algorithm.
  • Usage: Tracks set membership with path compression and union by rank.

3. Priority Queue

  • Structure: Heap (via heapq).
  • File: mst_algorithms.py.
  • Role: Selects minimum-weight edges in Prims algorithm.
  • Usage: Stores (weight, node, parent) tuples.

4. Animation Steps

  • Structure: List (animation_steps).
  • File: gui_algorithm_runner.py.
  • Role: Stores edges for animation.
  • Usage: Highlights MST construction.

5. GUI State

  • Structure: Instance variables (e.g., is_animation_running, large_graph).
  • Files: gui.py, gui_algorithm_runner.py, gui_weight_manager.py.
  • Role: Manages user interactions and updates.

Main Components of the Algorithms

The project implements optimized versions of Kruskals and Prims algorithms, supporting performance tracking.

1. Kruskals Algorithm (mst_algorithms.py)

  • Function: kruskal_mst(self)
  • Role: Computes the MST using sorted edges and Union-Find.
  • Steps:
    • Use preprocessed edge list (self.edges_list) from graph initialization.
    • Sort edges by weight.
    • Initialize parent as node indices and rank as zeros for Union-Find.
    • For each edge (u, v, w) in sorted order:
      • Use find() to check if u and v are in different sets.
      • If different, add edge to MST and unite sets with union().
    • Track runtime and edge comparisons as metrics.
    • If graph empty, return empty MST and error; else return MST edges and metrics (runtime, edge comparisons).
  • Supporting Functions:
    • find(self, parent, i): Finds set root with path compression.
    • union(self, parent, rank, x, y): Unites sets with rank optimization.

2. Prims Algorithm (mst_algorithms.py)

  • Function: prim_mst(self)
  • Role: Grows the MST from a starting node using a priority queue.
  • Steps:
    • Preprocess adjacency list (self.adj) from the graph.
    • Initialize visited set as empty.
    • Initialize heap with dummy edge (0, node 0, None), track nodes in heap with in_heap as a dictionary (node: min_weight).
    • While heap is not empty:
      • Pop minimum-weight edge (w, u, parent) from heap.
      • If u is already visited, skip.
      • Mark u as visited, remove u from in_heap.
      • If parent is not None, add edge (parent, u, w) to MST.
      • For each unvisited neighbor v of u with weight w_v: If v not in in_heap or w_v < in_heap[v], push edge (w_v, v, u) to heap and update in_heap[v] = w_v.
    • Track runtime and edge comparisons as metrics.
    • If not all nodes visited, note graph is unconnected.
    • Return MST edges and metrics (runtime, edge comparisons).
  • Supporting Functions: None additional.

3. Metrics Tracking

  • Function: reset_metrics(self)
  • Role: Resets runtime, edge comparisons, iterations before each run.
  • Function: get_metrics(self)
  • Role: Returns metrics for the comparison table.

Design of the User Interface

The GUI, built with PyQt5, prioritizes usability and clarity.

1. Layout

  • Window: 1000x600 pixels.
  • Structure:
    • Top (70%): Graph visualization area (GraphVisualizer).
    • Bottom (30%): Controls (left, 35%), comparison table (right, 65%).

2. Graph Visualization

  • Component: GraphVisualizer (Matplotlib canvas).
  • Features:
    • Shows nodes (blue), edges (lines), weights (labels).
    • Highlights MST edges (red) and vertices (light green) during animation.
    • Displays welcome message when idle.
  • Behavior: 0.25s/step animation for 27 nodes; instant for >27.

3. Controls

  • Elements:
    • Input fields: Nodes (1500), edges with weights (e.g., (1,2,19)).
    • Buttons: Generate Graph, Assign Weights, Run Algorithm, Clear.
    • Dropdown: Kruskals or Prims.
  • Behavior: Buttons disable during animation; error messages guide users.

4. Comparison Table

  • Component: TableManager.
  • Structure: Rows (MST Edges, Runtime, Total Weight, MST Weight, Total Nodes, Graph); Columns (Metric, Kruskals, Prims).
  • Features: Shows graph edges, algorithm results, weights; unified Total Nodes and Graph rows.

4. Analysis of Findings

  • Big O Notation:
    • Kruskals: O(E log E) O(V log V) for dense graphs (E V).
    • Prims: O((V + E) log V) O(V log V) with binary heap.
    • Prims outperforms Kruskals for large dense graphs (e.g., 500 nodes: 119.979 ms vs. 160.525 ms) due to efficient heap operations vs. edge sorting.
  • Performance for Immense Nodes: Prims scales better for V > 1000 in dense graphs; Kruskals may excel in sparse graphs (E << V).
  • Space Complexity:
    • Overall theoretical space, including graph storage, is O(V + E) (O(V) for dense graphs). This code measures auxiliary space: Kruskals Disjoint Set (O(V)), Prims Priority Queue peak size (O(V)). This O(V) measurement explains the linear growth observed.
    • Kruskals space grows slower (e.g., 7.922 KB at 500 nodes) vs. Prims (31.188 KB) due to smaller constant factors.

Usages

This MST Visualization Project has practical applications in various real-life scenarios, particularly in education and network optimization. Below are three examples:

1. Educational Tool for Algorithm Learning

  • The project can be used in computer science courses to teach students about MST algorithms. The visual animation of Kruskals and Prims algorithms helps students understand how edges are selected and how the MST is constructed, making abstract concepts more tangible.

2. Network Design Optimization

  • In telecommunications, the project can model the design of a cost-efficient network (e.g., laying cables between cities). By representing cities as nodes and cable costs as edge weights, the MST can minimize the total cost of connecting all cities, which is critical for budget planning.

3. Cluster Analysis in Data Science

  • The project can assist in clustering applications, such as grouping similar data points in machine learning. By treating data points as nodes and distances as edge weights, the MST can identify clusters by removing the largest edges, providing a visual tool for data scientists to explore hierarchical clustering.

Improvements

The project can be enhanced to provide more functionality, accessibility, and scalability. Below are some potential improvements:

1. Scaling and Removing Animation for Bigger and Denser Graphs

  • Currently, the application handles small to medium-sized graphs effectively, with animation enabled for educational visualization. However, for larger graphs with hundreds of thousands or millions of vertices, the app needs to scale efficiently. To achieve this, the visualization could be optimized by disabling animations for large graphs (e.g., graphs with more than 1,000 vertices) and instead displaying the final MST directly. Graph rendering could leverage more efficient libraries like Graphviz or D3.js (if web-based) to handle large-scale visualizations. On the backend, the MST computation could be optimized by using more efficient data structures, such as adjacency lists instead of NetworkX graphs, and implementing parallel processing for Kruskals algorithm (e.g., parallel edge sorting) or Prims algorithm (e.g., using a parallel priority queue). Additionally, the app could offload computation to a high-performance backend (e.g., using C++ or Rust for MST computation) and store large graphs in a database (e.g., Neo4j) to manage memory efficiently. These changes would enable the app to scale for massive graphs while maintaining usability.

2. Additional Fields in Table Manager

  • The comparison table can include more metrics for algorithm comparison, such as the number of edge comparisons or iterations (available in MSTAlgorithms.get_metrics()). Adding these fields would provide deeper insights into algorithm efficiency, especially for educational purposes.

3. User-Selected Starting Node for Prims

  • Currently, Prims algorithm starts from node 0. Allowing users to select the starting node (e.g., via an input field) would enable exploration of how the starting node affects the MST, which is valuable for understanding algorithm behavior in different scenarios.

4. Web Deployment with Django

  • Refactoring the project using a framework like Django would allow it to be deployed as a web application. This would make the tool accessible via a browser, enabling broader usage (e.g., in online classrooms). The GUI could be reimplemented using HTML/CSS/JavaScript, with the backend handling graph generation and MST computation.
